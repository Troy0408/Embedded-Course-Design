# 基于 STM32F103C8T6 的智能手表设计项目计划书

## 一、项目基本信息

| 项目 | 内容 |
| --- | --- |
| 课程设计题目 | 基于 STM32F103C8T6 的智能手表设计 |
| 项目类型 | 嵌入式系统综合课程设计 |
| 开发平台 | STM32F103C8T6、FreeRTOS |
| 主要外设 | OLED 显示屏、MPU6050、蓝牙模块、旋转编码器、电源管理模块 |
| 开发语言 | C 语言 |
| 开发工具 | Keil uVision、STM32CubeMX、串口调试助手、蓝牙上位机开发工具 |

本项目拟设计并实现一款基于 STM32F103C8T6 的嵌入式智能手表系统。系统以 STM32F103C8T6 为核心控制器，采用 FreeRTOS 实时操作系统完成多任务管理，集成 OLED 显示、MPU6050 姿态检测、时间显示、旋转编码器菜单交互、蓝牙通信和运动计步等功能。项目将覆盖硬件电路设计、外设驱动开发、I2C 总线通信、FreeRTOS 任务调度、数据同步、电源管理、系统联调与测试等内容。

## 二、项目概述

### 2.1 项目背景

随着可穿戴设备的发展，智能手表已成为嵌入式系统在日常生活中的典型应用之一。智能手表通常需要在有限的硬件资源和供电条件下完成信息显示、传感器采集、人机交互、无线通信和低功耗运行等任务，对嵌入式系统的软硬件协同设计能力要求较高。

STM32F103C8T6 具有成本低、资源适中、外设接口丰富、资料完善等特点，适合用于课程设计中的综合性嵌入式项目。FreeRTOS 能够提供任务调度、队列、互斥量和信号量等机制，有利于将显示、传感器采集、通信和交互等功能划分为相对独立的任务，提高系统结构的清晰性和可维护性。

本项目以智能手表为应用场景，综合训练 STM32 外设开发、FreeRTOS 多任务设计、I2C 通信、传感器数据处理、OLED 界面显示、蓝牙通信和系统调试能力，具有较强的课程实践意义。

### 2.2 项目目标

1. 完成基于 STM32F103C8T6 的智能手表硬件系统设计与搭建。
2. 完成 OLED、MPU6050、旋转编码器、蓝牙模块等主要外设的驱动开发。
3. 基于 FreeRTOS 搭建多任务系统，实现任务调度、数据同步和模块协作。
4. 实现时间显示、传感器数据显示、菜单切换、蓝牙通信和计步功能。
5. 完成系统集成、功能测试、稳定性测试和课程设计文档整理。
6. 在完成基本要求的基础上，力争完成旋转编码器菜单、蓝牙上位机和计步算法等扩展内容。

### 2.3 项目主要功能

本项目计划实现的主要功能如下：

1. 时间显示功能：在 OLED 屏幕上显示当前时间、运行状态等信息。
2. 传感器数据采集功能：通过 I2C 读取 MPU6050 的加速度和角速度数据。
3. OLED 界面显示功能：实现时间页面、传感器数据页面、运动数据页面和系统状态页面。
4. 多任务管理功能：使用 FreeRTOS 管理显示、采集、通信、菜单和算法等任务。
5. 菜单交互功能：通过旋转编码器实现页面切换、菜单选择和参数设置。
6. 蓝牙通信功能：通过 HC-05 或 HM-10 与手机或上位机进行数据传输。
7. 运动计步功能：基于 MPU6050 加速度数据实现简单计步算法和运动数据记录。
8. 电源管理功能：采用锂电池供电，结合 LDO 稳压模块为系统提供稳定电源。

### 2.4 项目预期成果

项目完成后，预期形成一套可演示的智能手表原型系统。系统能够稳定显示时间和传感器数据，能够通过旋转编码器切换菜单页面，能够通过蓝牙向手机或上位机发送运动数据，并能够完成基本计步演示。同时提交项目计划书、硬件原理图、程序源码、测试记录、课程设计报告和答辩 PPT 等材料。

### 3.2 性能需求

1. OLED 时间显示刷新周期不大于 1 s，界面切换响应时间不大于 300 ms。
2. MPU6050 数据采样周期控制在 20 ms 至 100 ms 范围内，满足姿态检测和计步算法需求。
3. 蓝牙通信能够稳定发送时间、传感器数据和步数数据，常规通信丢包率保持在可接受范围内。
4. FreeRTOS 系统应连续运行不少于 2 小时无明显卡死、重启或任务异常。
5. 系统供电电压保持稳定，主控和外设工作电压满足器件要求。
6. 程序结构清晰，任务划分合理，便于后续扩展和调试。

### 3.3 硬件需求

硬件系统以 STM32F103C8T6 最小系统板为核心，外接 OLED 显示屏、MPU6050、蓝牙模块、旋转编码器、电源管理模块和锂电池。OLED 和 MPU6050 采用 I2C 接口，蓝牙模块采用 UART 接口，旋转编码器使用 GPIO 外部中断或轮询方式读取。电源部分采用 3.7V 锂电池供电，通过充电模块和 LDO 稳压模块为系统提供稳定的 3.3V 工作电压。

### 3.4 软件需求

软件系统采用 C 语言开发，基于 STM32 标准外设库或 HAL 库完成底层驱动开发，并移植 FreeRTOS 实现任务管理。软件主要包括系统初始化、FreeRTOS 任务创建、OLED 驱动、MPU6050 驱动、编码器驱动、蓝牙串口通信、时间管理、菜单管理、计步算法和测试调试模块。

### 3.5 人机交互需求

系统通过 OLED 屏幕向用户显示时间、传感器数据、运动步数和系统状态。用户可通过旋转编码器进行页面切换和菜单选择。蓝牙上位机用于接收手表端发送的数据，并可显示时间、姿态数据、步数和连接状态。整体交互方式应简单直观，适合课程答辩现场演示。

## 四、总体方案设计

### 4.1 系统总体架构

系统总体采用“主控处理器 + 传感器采集 + 本地显示 + 人机输入 + 蓝牙通信 + 电源管理”的结构。STM32F103C8T6 作为核心控制器，负责外设数据采集、任务调度、界面刷新、蓝牙数据收发和算法处理。OLED 用于本地显示，MPU6050 用于姿态和运动数据采集，旋转编码器用于菜单交互，蓝牙模块用于与手机或上位机进行数据同步。

系统数据流关系如下：

1. MPU6050 通过 I2C 将加速度和角速度数据传输给 STM32。
2. 计步算法任务读取传感器数据，计算步数和运动状态。
3. 时间管理任务维护系统时间，并向显示任务提供时间数据。
4. 菜单交互任务根据旋转编码器输入更新当前页面状态。
5. OLED 显示任务根据当前页面显示时间、传感器数据、步数或系统状态。
6. 蓝牙通信任务将时间、姿态、步数等数据发送至手机或上位机。
7. 系统监测任务检查任务运行状态、电源状态和通信状态。

### 4.2 硬件总体方案

| 模块 | 主要器件 | 接口方式 | 设计说明 |
| --- | --- | --- | --- |
| 主控模块 | STM32F103C8T6 最小系统板 | GPIO、I2C、UART | 负责系统控制、任务调度和数据处理 |
| 显示模块 | 1.3 寸 I2C OLED | I2C | 显示时间、菜单、传感器和运动数据 |
| 姿态检测模块 | MPU6050 | I2C | 采集三轴加速度和三轴角速度 |
| 蓝牙通信模块 | HC-05 或 HM-10 | UART | 与手机或上位机通信 |
| 输入模块 | 旋转编码器 | GPIO/EXTI | 实现菜单切换和选择 |
| 电源模块 | 锂电池、Type-C 充电模块、LDO | 电源接口 | 提供稳定 3.3V 电源 |
| 调试模块 | SWD、串口 | SWD/UART | 用于程序下载和调试输出 |

OLED 和 MPU6050 均使用 I2C 总线，为避免地址冲突和总线异常，设计时将确认两者 I2C 地址，并在程序中增加总线错误处理机制。蓝牙模块使用独立 UART 接口，便于调试和数据发送。若后期进度允许，将根据原理图完成 PCB 绘制和打样；若时间不足，则采用最小系统板、洞洞板或面包板完成实物连接。

### 4.3 软件总体方案

软件系统分为底层驱动层、中间服务层和应用任务层。

1. 底层驱动层：包括 GPIO、I2C、UART、SysTick、OLED、MPU6050、编码器和电源检测等驱动。
2. 中间服务层：包括 FreeRTOS 队列、互斥量、事件标志组、菜单管理、数据缓存和通信协议封装。
3. 应用任务层：包括时间管理任务、OLED 显示任务、传感器采集任务、菜单交互任务、蓝牙通信任务、计步算法任务和系统监测任务。

系统采用模块化设计，各功能模块通过统一的数据结构交换信息，减少模块之间的直接耦合。对于 OLED 显示和 I2C 访问等共享资源，使用互斥量进行保护，避免多任务同时访问造成冲突。

### 4.4 FreeRTOS 任务划分方案

| 任务名称 | 优先级 | 任务周期 | 主要功能 | 通信方式 |
| --- | --- | --- | --- | --- |
| 时间管理任务 | 中 | 1000 ms | 维护系统时间，更新时间数据 | 全局时间结构体、互斥量 |
| OLED 显示任务 | 中 | 100 ms | 根据当前页面刷新 OLED 显示内容 | 队列、互斥量 |
| MPU6050 数据采集任务 | 高 | 20-50 ms | 读取加速度和角速度数据，完成基础滤波 | 队列、共享数据缓存 |
| 菜单交互任务 | 高 | 20 ms | 读取旋转编码器输入，切换菜单和页面 | 事件标志组、队列 |
| 蓝牙通信任务 | 中 | 200-500 ms | 发送时间、姿态和步数数据，接收控制命令 | UART、队列 |
| 计步算法任务 | 中 | 100 ms | 分析加速度数据，统计步数和运动状态 | 队列、共享运动数据 |
| 系统监测任务 | 低 | 1000 ms | 检查任务运行状态、电源状态和异常标志 | 事件标志组 |

任务优先级设置原则为：传感器采集和菜单输入响应优先级较高，保证数据及时性和交互响应；OLED 显示和蓝牙通信为中等优先级，保证界面和通信稳定；系统监测任务优先级较低，用于后台状态检查。

### 4.5 各模块数据流关系

| 数据来源 | 数据内容 | 接收模块 | 用途 |
| --- | --- | --- | --- |
| 时间管理任务 | 当前时间、运行秒数 | OLED 显示任务、蓝牙通信任务 | 时间显示和数据同步 |
| MPU6050 任务 | 加速度、角速度、姿态状态 | OLED 显示任务、计步算法任务、蓝牙通信任务 | 姿态显示和运动分析 |
| 菜单交互任务 | 当前页面、菜单选择状态 | OLED 显示任务、系统控制模块 | 页面切换和参数设置 |
| 计步算法任务 | 步数、运动状态、运动记录 | OLED 显示任务、蓝牙通信任务 | 运动数据显示和同步 |
| 蓝牙通信任务 | 手机端命令、同步请求 | 时间管理任务、菜单模块 | 时间校准和状态查询 |
| 系统监测任务 | 电源状态、异常状态 | OLED 显示任务、蓝牙通信任务 | 状态提示和异常反馈 |

### 7.2 成本估算

基础实现所需器件包括 STM32F103C8T6 最小系统板、OLED、MPU6050、电源模块、锂电池、杜邦线、排针、阻容元件和面包板等，估算成本约为：

**18 + 25 + 8 + 5 + 20 + 3 + 8 + 8 + 3 + 10 = 108 元。**

扩展实现所需器件包括蓝牙模块、旋转编码器、外壳材料、蜂鸣器或振动马达等，估算增加成本约为：

**18 + 3 + 20 + 5 = 46 元。**

若进行 PCB 打样，则增加约 **50 元**。因此：

| 成本类别 | 估算金额 |
| --- | --- |
| 基础实现成本 | 约 108 元 |
| 扩展实现成本 | 约 154 元 |
| 含 PCB 打样的完整成本 | 约 204 元 |

以上价格根据常见网购平台和实验室常用器件价格估算，实际采购时可优先使用实验室已有物料，以降低项目成本。

## 九、技术路线

本项目的技术路线按照“硬件平台搭建、底层驱动调试、实时系统移植、功能模块开发、系统集成测试”的顺序推进。

### 9.1 实现流程

1. 搭建 STM32F103C8T6 最小系统，完成下载调试环境配置。
2. 连接 OLED 和 MPU6050，确认 I2C 地址和通信线路。
3. 完成 OLED I2C 通信测试，实现字符、数字和页面显示。
4. 完成 MPU6050 初始化和数据读取，获取加速度和角速度原始数据。
5. 移植 FreeRTOS，建立多任务工程框架。
6. 创建时间管理任务，实现系统运行时间和时间显示。
7. 创建 OLED 显示任务，实现时间页面、传感器页面和运动数据页面。
8. 创建 MPU6050 数据采集任务，实现周期采样和数据滤波。
9. 接入旋转编码器，实现菜单切换、页面选择和按压确认。
10. 接入蓝牙模块，实现 STM32 与手机或 PC 上位机之间的数据发送。
11. 设计简单通信协议，例如以 `TIME`、`IMU`、`STEP` 作为数据帧类型。
12. 基于加速度模值变化实现计步算法，完成步数统计和运动状态判断。
13. 对各任务进行联调，优化任务周期、优先级和共享资源访问方式。
14. 进行稳定性、显示效果、蓝牙通信、计步准确性和功耗测试。
15. 整理课程设计报告、测试记录、答辩 PPT 和演示视频或图片。

### 9.2 文字流程图

系统上电启动  
-> 初始化时钟、GPIO、I2C、UART 和外设  
-> 初始化 OLED、MPU6050、蓝牙模块和旋转编码器  
-> 创建 FreeRTOS 任务、队列、互斥量和事件标志组  
-> 启动 FreeRTOS 调度器  
-> MPU6050 任务周期采集数据  
-> 计步算法任务处理运动数据  
-> 菜单任务读取编码器输入并更新页面状态  
-> OLED 任务根据页面状态刷新显示  
-> 蓝牙任务向上位机发送时间、姿态和步数数据  
-> 系统监测任务检查运行状态  
-> 持续循环运行并响应用户操作

## 十、风险分析与解决方案

| 风险点 | 可能原因 | 影响 | 解决方案 |
| --- | --- | --- | --- |
| I2C 多设备通信冲突 | OLED 和 MPU6050 共用 I2C，总线时序异常或地址配置错误 | OLED 无法显示或 MPU6050 数据读取失败 | 确认设备地址；降低 I2C 速率；增加总线互斥量；分阶段单独测试设备 |
| OLED 显示刷新卡顿 | 刷新频率过高、整屏刷新耗时较长、显示任务优先级不合理 | 界面响应慢，影响菜单体验 | 采用局部刷新或降低刷新频率；优化显示缓存；合理设置显示任务周期 |
| FreeRTOS 任务优先级设置不合理 | 高优先级任务占用 CPU 时间过长，低优先级任务得不到执行 | 系统卡顿、通信延迟、显示不稳定 | 对任务执行时间进行测试；在循环中加入延时；调整任务优先级和周期 |
| 任务间共享资源冲突 | 多任务同时访问 I2C、OLED 缓存或全局数据 | 数据异常、显示错乱或程序死锁 | 使用互斥量保护共享资源；统一数据访问接口；避免在中断中执行耗时操作 |
| MPU6050 数据波动较大 | 传感器噪声、手持抖动、安装不稳定 | 姿态显示和计步算法误差增大 | 增加滑动平均或低通滤波；设置有效阈值；固定模块安装位置 |
| 蓝牙连接不稳定 | 波特率配置错误、供电不足、距离过远或模块兼容性问题 | 数据无法同步或频繁断连 | 统一波特率；保证电源稳定；缩短测试距离；增加重发和连接状态提示 |
| 计步算法误差较大 | 阈值设置不合理、动作干扰、采样频率不稳定 | 步数统计不准确 | 使用加速度模值；设置动态阈值和时间窗口；通过实际走步测试修正参数 |
| 电源供电不稳定 | 锂电池电压下降、LDO 输出能力不足、模块瞬时电流较大 | 系统重启、蓝牙掉线或传感器异常 | 增加滤波电容；检查模块电流需求；使用稳定 LDO；调试阶段可先用 USB 供电 |
| 进度延期风险 | 硬件到货延迟、联调问题复杂、文档整理滞后 | 影响最终验收和答辩质量 | 提前采购关键器件；优先完成基本功能；扩展功能分级实现；文档随开发同步整理 |

## 十三、总结

本项目围绕“基于 STM32F103C8T6 的智能手表设计”展开，计划完成智能手表硬件平台搭建、FreeRTOS 多任务系统设计、OLED 界面显示、MPU6050 数据采集、旋转编码器菜单交互、蓝牙通信和计步算法等内容。项目既包含 STM32 基础外设开发，也涉及实时操作系统、任务调度、总线通信、传感器数据处理和系统联调，能够较全面地训练嵌入式系统课程设计所需的综合实践能力。

在实施过程中，小组将围绕 2026 年 7 月 9 日验收节点倒排计划，从需求分析、方案设计、物料准备、驱动开发到系统集成和测试验收快速推进。项目目标是在保证基本功能稳定可靠的基础上，重点完成菜单交互、蓝牙通信和计步算法扩展功能，并根据剩余时间尝试 PCB 设计或外观结构优化。通过本项目的设计与实现，小组成员能够进一步理解嵌入式系统软硬件协同开发流程，提高问题分析、模块调试、团队协作和工程文档编写能力，为后续更复杂的嵌入式项目开发打下基础。
